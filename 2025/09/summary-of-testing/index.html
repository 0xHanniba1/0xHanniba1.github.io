测试回顾

Posted at 2025-09-01

功能测试：站在用户角度 → “新功能能不能用”。

接口测试：站在系统角度 → “系统对系统的数据交互对不对”。

回归测试：站在全局角度 → “新功能上线，旧功能还稳不稳”。

🗺️ 软件测试方法地图

1. 按照“测试阶段”分类（从小到大）

就像建一栋楼，要从 砖头 → 房间 → 整栋楼 → 交付业主。

  • 单元测试（Unit Test）
    • 检查最小的功能单元（比如一个函数、一个接口）。
    • 类比：看一块砖头是不是结实。
    • 通常由开发写。
  • 集成测试（Integration Test）
    • 检查模块之间能不能合作。
    • 类比：把砖头砌成一面墙，看看是不是牢固。
    • 例子：优惠券模块 + 支付模块能否顺畅衔接。
  • 系统测试（System Test）
    • 检查整个系统作为一个整体是不是能跑通。
    • 类比：房子整体能不能住人。
    • 功能测试、接口测试、性能测试都在这里。
  • 验收测试（Acceptance Test）
    • 给用户或产品验收，确认系统符合需求。
    • 类比：业主收房，看装修是不是达标。

2. 按照“测试目的”分类（测什么）

像体检时，不同检查项目对应不同目标。

  • 功能测试：测业务逻辑对不对。
  • 接口测试：测模块/系统交互是否正确。
  • 性能测试：测响应速度、并发量（房子能不能抗住 100 个人同时住）。
  • 安全测试：测有没有漏洞（门锁结实不结实）。
  • 回归测试：改动后确认旧功能是否正常。

3. 按照“测试方式”分类（怎么测）

  • 手工测试：人工操作、观察结果。
  • 自动化测试：写脚本执行测试，提高效率。
  • 黑盒测试：只关心输入和输出，不看内部代码（像按电梯按钮，只管能不能到楼层）。
  • 白盒测试：了解代码逻辑，设计用例（像看电梯内部电路怎么走）。

👉 这样，你就能从三个维度理解测试方法：

  • 阶段：从小到大
  • 目的：从“做对”到“做快/做安全”
  • 方式：从人工到自动

二：软件测试流程

软件测试流程，可以类比成 做一次大型体检：

  1. 需求分析：先问清楚“要查什么项目” → 就像医生先了解你的症状和病史。
  2. 测试计划：确定“怎么查、查哪些” → 安排体检套餐。
  3. 测试用例设计：写下每一步具体操作 → 比如“抽血 → 验血 → 看结果”。
  4. 测试执行：按照用例一步步操作 → 真的去抽血、做CT。
  5. 缺陷管理：发现问题要记录并反馈 → 就像报告里写“血糖偏高”。
  6. 回归测试：治疗或调整后再复查 → 看之前的问题是不是解决了。
  7. 测试报告：总结体检结果 → 提交完整的结论给用户/产品经理。

在缺陷管理中，通常会标记 严重程度（Severity） 和 优先级（Priority）。

  • 严重程度：问题有多严重？
    • 致命（比如：支付不成功）
    • 严重（比如：优惠券逻辑错误）
    • 一般（比如：提示信息错别字）
  • 优先级：要多快修？
    • 高（必须立刻修）
    • 中（可以下个版本修）
    • 低（不影响核心功能，可以延期）

三：测试用例设计

测试用例（Test Case），其实就是一份 “操作手册”，告诉测试人员：

  • 要输入什么？
  • 要怎么操作？
  • 期望结果是什么？

常见的方法有：

  1. 等价类划分
    • 把输入分成“有效”和“无效”区间，各挑一个代表。
    • 类比：检查水果是不是坏的，不用一个个切开，挑代表性样本就行。
  2. 边界值分析
    • 错误往往出现在“临界点”。
    • 类比：杯子装水，9分满正常，10分满可能溢出来 → 所以要特别测“边界”。
  3. 因果图 / 判定表
    • 用逻辑关系分析复杂输入组合。
    • 类比：开锁要同时满足“钥匙正确 + 密码正确”。
  4. 错误推测法
    • 基于经验，猜测容易出错的地方。
    • 类比：猜新手司机可能会忘记打方向灯，就重点检查。
